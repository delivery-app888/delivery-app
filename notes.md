分析項目にある時間帯分析などの時間表示をしているグラフについて、現在3時間区切りがメインだが、詳細がわからないので全て1時間区切りに変更する。1時間区切りにすると画面幅に収まりきらないため、横スクロールバーをつけてスクロールできるようにする。

## ✅ 20. 高単価配達ヒートマップの条件とピンタッチ遷移
高単価配達ヒートマップは1000円以上の配達を対象とする。ただし、ピックゴーの配達は高単価配達ヒートマップには記載しない。高単価ヒートマップのピンをタッチすると、その配達の詳細ページに移動できるようにする。

## ✅ 21. トーストの全パターン一覧の確認
平均ベースなどのトーストがいろんな種類があるが、トーストにはどんな種類があるのか、すべてのパターンを洗い出す。

## ✅ 22. PickGo手数料の自動反映システム
PickGoは入金する際に手数料がかかる。総額が25万円以上の時は10%、25万円未満の時は15%。配達を受注して配達完了する時に、PickGoを選んだ時に入れた金額にこの手数料を反映する。設定画面にPickGo手数料率のトグルを作る（25万円未満15% / 25万円以上10%）。デフォルトは15%（多くの人は25万円未満まで引き出すため）。手数料が含まれたままだと分析結果の正確性が落ちてしまうので、手数料を差し引いた金額で記録する。

## ✅ 23. 配達詳細メモの当日以外も編集可能化
配達の詳細メモに関しては、当日以外も編集可能にする。メモ機能は分析の数値には関係ないが、高単価配達ヒートマップに閉店情報や考察などを書き足していくことによって、高単価の配達案件を受注できる可能性を最大化させるため。

## ✅ 24. 色分けの統一（ヒートマップ・エリア分析等）
現在、配達ヒートマップや高単価配達ヒートマップ、エリア別分析などの色分けをしている項目は、ページごとに色の種類が違う。ページごとに内容を理解しないといけないので、見間違いや勘違いが発生する可能性がある。色は統一する。グラデーションではなく単色でわかりやすくする。基本的には数値が高いものを赤、数値が低いものを青、間はコントラストが出る色にする。すべての項目を統一するため、色分けを使っている分析項目をすべて洗い出す必要がある。

## ✅ 25. 「このペースでの着地予想」の計算方法の確認と修正
メイン画面に表示されている「このペースでの着地予想」の計算方法について、コードではなくわかりやすく日本語で計算方法を説明してほしい。
→ 変更前: 1日の稼働時間を固定10時間と仮定して残り時間を計算していた。副業など短時間稼働の人には不正確。初めて使う人はこの前提を知らないため、誤解を招く。
→ 変更後: 過去のすべてのログから各日の最後のオフライン時刻を収集し、加重平均で「予測オフライン時刻」を算出。今日と同じ曜日のデータは重み2倍、直近のデータほど重みが大きい。予測オフライン時刻−現在時刻＝残り稼働時間。今日の時給ペース×残り時間＋現在の売上＝着地予測。データ3日未満の場合は着地予測を非表示にする。

## ✅ 26. 配達完了ボタンの2度押し防止
受注ボタンを1回押すたびに、配達完了ボタンは1回しか押せなくなる。たまに配達完了を2度押ししているのか、同じ配達終了時刻の配達が2つ続いている場合がある。
→ 調査の結果、配達完了ボタン自体は受注1回につき1回しか押せない設計になっていた。原因は報酬入力画面のOKボタンの2度押し。OKボタン押下後にGPS取得を待つasync処理中に再度タップすると、同じcurrentOrderTimeで2件保存される可能性があった。rwSavingフラグを追加し、OK押下から保存完了までボタンを無効化（「保存中...」表示）することで解消。実稼働での検証待ち。

## ✅ 27. ゴールデンタイムの時給の計算根拠の確認と修正
ゴールデンタイムとメイン画面に書かれている曜日と時間帯があるが、あの時給は何を根拠にどのような計算をされているのか確認する。
→ 変更前: 各時間帯の報酬合計÷配達所要時間合計で時給を計算していた。待機時間が含まれないため、実際よりかなり高い時給が表示されていた。
→ 変更後: 各時間帯の売上合計÷その時間帯に配達があった稼働日数で計算。例えば水曜14時台の時給＝過去の水曜14時台の売上合計÷水曜で14時台に配達があった日数。同曜日のデータを優先（3日以上あれば使用）、不足時は全曜日にフォールバック。データ3日未満は非表示。曜日データ使用時は「水曜のゴールデンタイム」のように曜日も表示。判定基準は変更なし（平均の1.15倍以上＝ゴールデン、0.85倍以下＝まったり）。

## ✅ 28. PickGo報酬の手数料引き後の金額を絶対参照

## ✅ 29. 日跨ぎオンラインの対応
日を跨いでオンラインの可能性はあるので、日を跨いだときに自動的にオフラインにするのはやめてほしい。これを解決するためにはどのような方法があるか検討する。
→ 案Aは日跨ぎ時にセッションを0:00で分割する方式だが、1つの稼働が2日に分かれるのは直感に反するため不採用。
→ 当初は案B（オフラインまで1つの稼働日として保持）を採用したが、待機時間が26時間と表示される問題が発生。案Aベースに変更：0:00に自動でセッション・休憩・地蔵を23:59:59で分割し、前日データを保存。新しい日のデータを作成し0:00からセッションを自動開始。配達が日を跨いだ場合は受注した日に帰属（前日のログに保存）。編集可能条件は「完了後60分以内」に変更（改ざん防止と日跨ぎ時の修正を両立）。

## ✅ 30. 効率化ポイントの数値表示の意味が不明

## ✅ 42. 効率化ポイントからPickGo以外の分担価100円以上の配達を除外
効率化ポイントには、分担価100円以上のPickGo以外の配達は含めない。理由：PickGo以外の分担金100円以上は特定の条件下でないと発生しないため（閉店時間超過でマッチングしていない注文、受注後1時間経過でマッチングしていない注文、大雨・大雪等の異常気象、店舗に配達員未マッチングの調理完了商品が多数滞留している状況など）。PickGoは時間テーブルで配達料金が固定されているため、分担価100円以上を含んでも問題ない。
→ 3点を修正：(1) PickGo以外の分給100円/分以上の配達を効率化ポイントから除外、(2) 所要時間3分未満の配達も除外（ボタン押し忘れ修正等による異常データ防止）、(3) 分給の計算を平均値から中央値に変更（外れ値の影響を排除し、普段の実力を正確に反映）。全5ルール＋メイン画面サマリーに適用。
→ 追加で以下2箇所も中央値に変更：(4) 配達単価（pastAvgUnit/avgUnitForGuide）— 残り配達件数の計算、自動評価（good/bad判定）、配達後フィードバックの精度向上、(5) 効率化ポイント画面の分給表示 — ラベルも「平均分給」→「中央分給」に変更し分析ロジックと統一。平均単価分析グラフ・曜日別売上は合計金額との整合性のため平均値のまま据え置き。

## ⬜ 43. 次期方針：記録精度を先に固めてから分析精度を上げる
今後の実装順は、先に「記録の正確さ」を固め、その後に「分析の賢さ」を上げる。優先順位は以下。
1. 店舗到着・店舗出発ボタンを追加し、待機5分以上または調理待ちキャンセルを店舗待機ピンとして残す。
2. キャンセルは「店舗未到着キャンセル」と「調理待ちキャンセル」に分ける。店舗未到着キャンセルは待ち時間マップに載せない。
3. オンライン・オフライン時間を配達履歴の一番上に表示し、翌日23:59まで編集可能にする。
4. 配達ごとのインセンティブと日次インセンティブも履歴から編集可能にする。編集期限は稼働時間と同じく翌日23:59までで統一する。
5. 自動オフラインは設定式にする。初期値はOFF。1時間/2時間/3時間から選択し、最後の操作から指定時間が経過したら自動終了候補にする。アプリ再起動時の補正も入れる。
6. 記録精度が固まった後、曜日別×時間別の時給表、着地予測の平均稼働時間ベース化、効率化ポイントの「会社を避ける」から「報酬・分単価の基準を出す」形式への変更、雨量データ強化、年間売上サマリーを実装する。

## ✅ 44. CSVテキストダウンロード機能
設定画面にCSVテキストをダウンロードする機能を追加する。目的は、実データをCodexに渡して追加分析や次の機能検証に使うこと。
→ 1つのCSVに `day_summary` / `session` / `break` / `jizo` / `delivery` / `daily_incentive` の行を入れる。GPS座標、天候API、雨量、メモ、インセンティブも含める。将来の店舗到着・店舗出発・キャンセル種別用の列も先に用意しておく。

## ✅ 45. 店舗到着・店舗出発・キャンセル2種類化
受注後の下部固定ボタンを段階式に変更する。通常時は「受注」、受注後は「店舗到着」と「未到着キャンセル」、店舗到着後は「店舗出発」と「調理待ちキャンセル」、店舗出発後は「配達完了」を表示する。
→ `storeArrivalTime` / `storeDepartTime` / `storeLat` / `storeLng` / `storeWeather` / `cancelType` を追加。キャンセルは `before_store`（店舗未到着）と `store_wait`（調理待ち）で保存。調理待ちキャンセル、または店舗待機5分以上の配達は店舗待機マップに店舗ピンとして表示する。過去データは migrate で null 補完する。

## ✅ 46. 店舗待機マップの専用画面化
注文ヒートマップに店舗待機ピンを混ぜると、「受注が多い場所」と「待たされる店」の意味が混ざるため、専用の「店舗待機マップ」に分離する。
→ 注文ヒートマップは受注地点のみを表示。店舗待機マップは待機5分以上、10分以上、15分以上/調理待ちキャンセルを色分けして表示する。期間・時間帯・曜日・会社・天候フィルターは注文ヒートマップと同じ操作に揃える。

## ✅ 47. 配達履歴でオンライン・オフライン時間を修正
配達履歴の日別カードを開いた一番上に、オンライン時刻・オフライン時刻・稼働時間を表示する。修正可能期間は対象日の翌日23:59まで。
→ `sessions` を編集対象にし、当日・前日までのオンライン開始/オフライン終了を履歴から修正可能にした。本日オンライン中のデータも履歴に表示し、稼働中の場合はオフライン欄を「稼働中」と表示する。

## ✅ 48. 配達履歴でインセンティブを修正
配達ごとのインセンティブと日次インセンティブを、配達履歴から修正できるようにする。修正可能期間は稼働時間と同じく翌日23:59まで。
→ 配達詳細で配達インセンティブを編集・保存可能にした。日別カード内の日次インセンティブも追加・修正・削除できるようにし、保存後は日別売上・時給・CSV出力に反映される。

## ✅ 49. 月間・年間売上集計
月間は毎月1日から月末まで、年間は1月1日から12月31日までで集計する。会社別に表示し、一番上は全社合計にする。
→ 分析メニューに「売上集計」を追加。月間/年間を切り替え、前月・翌月、前年・翌年を移動できる。全社合計を一番上に表示し、Uber Eats/出前館/menu/Rocket Now/PickGoごとの売上、件数、インセンティブ内訳を表示する。

## ✅ 50. 実測雨量データの管理強化
気象コンディション分析で、取得した実際の雨量を日別データとして管理する。配達内のAPI値だけに散らすと、日別・期間別の雨量集計が弱くなる。
→ 日別ログに `weatherSamples` を追加し、受注時・店舗到着時・店舗出発補完時の天候API結果を保存する。既存データは配達の `apiWeather` / `storeWeather` からサンプルを復元する。CSVには `weather_sample` 行、日別平均雨量、最大雨量、取得回数、雨量レベルを出力する。デイリーレポートと気象コンディション分析にも実測雨量の取得数・最大雨量・雨あり日を表示する。

## ✅ 51. 本日の着地点予測を平均オンライン時間ベースに変更
着地点予測は、過去のオフライン時刻ではなく「過去の平均オンライン稼働時間」を今日のオンライン開始時刻に足して計算する。朝8時から開始し、平均オンライン時間が8時間なら16時着地で予測する。
→ `pacePredict` を過去の終了時刻平均から、過去セッションのオンライン継続時間平均に変更。予測カードには、想定する平均オンライン時間と終了想定時刻も表示する。

## ✅ 52. 曜日×時間の時給表
曜日別・時間別の時給表を作り、働く時間や休みを決める目安にする。単純な売上合計ではなく、その時間に実際オンラインだった時間から休憩を差し引いて時給を算出する。
→ 分析メニューに「曜日×時間 時給表」を追加。1週間/1ヶ月/半年/全期間で切り替え、曜日7列×24時間の表で時給を表示する。件数が少ないセルは灰色で表示し、今の時間の目安時給と最高実績の曜日・時間も表示する。

## ✅ 53. 効率化ポイントを数値基準型に変更
会社や時間帯を一律に避ける助言は、他社単価や当日の状況で逆効果になる可能性があるため、受注判断に使える数値基準へ寄せる。
→ 効率化ポイントのルール表示を「避ける」から「この条件では¥○/分以上だけ取る」「報酬¥○以上、または¥○/分以上を基準にする」に変更。詳細カードも比較対象を「取る基準」として表示し、会社名ベースではなく単価・分単価ベースの助言に寄せた。

## ✅ 54. 自動オフライン機能
設定で OFF / 1時間 / 2時間 / 3時間 を選び、オンライン中に無操作が続いた場合は自動でオフラインにする。PWAはバックグラウンドで常時動けないため、前面表示中の自動終了に加えて、復帰時・再起動時の補正も入れる。
→ 設定画面に自動オフライン設定を追加。オンライン開始時と操作時に `currentLastActivityAt` を保存し、最後の操作から設定時間を超えたら `sessions` / `breaks` / `jizoSessions` をその時刻で閉じる。受注中は記録保護のため自動終了しない。アプリ復帰時・再起動時にも、最後の操作時刻＋設定時間でセッションを補正する。

## ✅ 55. Rocket Now追加報酬率設定
PickGo手数料設定と同じように、Rocket Nowのリワード追加報酬率を設定できるようにする。選択肢は、追加報酬なし、グリーン10%、ブルー15%、パープル20%、ゴールド25%、ゴールドプラス30%。
→ 設定画面にRocket Now追加報酬を追加。Rocket Nowで配達報酬を入力した場合のみ、選択中の追加報酬率を入力金額に上乗せして保存する。報酬入力画面には反映後金額を表示し、履歴詳細には基本金額と適用率を表示する。CSVには `raw_reward` と `rocket_bonus_rate` を出力する。

## ✅ 56. 個人情報保護のためのPWAセキュリティ補強
位置情報を含む外部API URLが端末内キャッシュに残るリスクを下げる。
→ Service Workerは同一オリジンのアプリファイルだけをキャッシュし、Open-Meteo、Nominatim、OpenStreetMapタイルなど外部リクエストはキャッシュ対象外にした。外部APIへ送る緯度経度はローカル保存値とは別に約100m単位へ丸め、fetchは `no-store` / `no-referrer` を指定。Leafletポップアップに出す文字列もHTMLエスケープして、保存データ経由の表示崩れやXSSリスクを下げた。依存パッケージは `npm audit` で0件になるまで更新した。
