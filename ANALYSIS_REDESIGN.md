# 分析機能 再設計メモ

作成日: 2026-05-30

## 目的

このアプリの分析機能は、単にグラフや集計項目を増やすことではなく、配達員の売上を上げる判断に使えることを目的にする。

特に重要なのは、配達員が現場で次の判断を直感的にできること。

- 今、続けるか / 休むか / 移動するか
- この案件を取るか / 捨てるか
- どこで待つか
- どの時間に稼働するか
- どの会社を優先して見るか
- どの店舗やエリアを避けるか

分析は「見て終わり」ではなく、「次の配達に落とし込める」形にする。

## 現在存在している主な分析

現在のアプリには、以下の分析・関連画面が存在している。

- デイリーレポート
- 売上集計
- 効率化ポイント
- 注文ヒートマップ
- 店舗待機マップ
- 高単価配達マップ
- 時間帯分析
- 曜日 x 時間 時給表
- エリア別分析
- 気象コンディション分析
- 曜日分析
- 会社別分析
- 天候分析
- 推移（週次/月次）
- 平均単価

これらは個別には価値があるが、画面数が増えすぎるとユーザーが「何を見れば行動が変わるのか」を判断しづらくなる。

今後は、分析項目を増やすより、目的別に統合していく方がよい。

## 現状の課題

### 1. データから作れる分析が先行している

時間帯、曜日、会社、天候、エリア、地図など、記録データから作れる分析が増えている。

しかし、売上を上げるためには「その結果、何を変えるべきか」まで出す必要がある。

### 2. 平均だけでは現場判断に弱い

単純な平均時給や平均単価は参考になるが、実際の受注判断はもっと条件付きで行われる。

例:

- ピークタイムのロング案件は、単価が良くても機会損失になることがある
- オフピークでは、分単価がボーダー以上ならロングも受ける
- 調理待ちしやすい店舗は、ピークでもオフピークでも基本的に避ける
- 会社別の優劣は、時間帯・季節・ミッション状況で変わる

### 3. 地図分析の意味が分散している

注文ヒートマップ、高単価配達マップ、店舗待機マップ、エリア別分析はそれぞれ意味が違う。

ただしユーザーは最終的に「どこで待つか」「どこを避けるか」を知りたい。

地図系は、別画面を増やすより、意味ごとのレイヤーとして整理した方が直感的になりやすい。

## 配達員の実務知見

以下は実際の配達経験に基づく重要な前提。

### 受注判断で見ている情報

受注判断の瞬間には、以下を総合的に見ている。

- 報酬
- 距離
- 店名
- 届け先方向
- 注文数
- 予想時間
- 会社ごとの差
- 稼働している時間帯

特に重要なのは、届け先方向が「基点となる場所に戻れる方向かどうか」。

報酬そのものも重要だが、実際には分単価が重要。

### ピークタイムとロング案件

暫定ピーク定義:

- 昼ピーク: 11:00-13:00
- 夜ピーク: 18:00-21:00
- 深夜ピーク: 01:00-04:00

ただし、深夜ピークは地域によって存在しない場合がある。

ピークタイムは単なる時刻ではなく、「人が食事をする生活リズム」と「エリア特性」の掛け合わせで考える。

- 昼ピークは昼食需要
- 夜ピークは夕食需要
- 深夜ピークは夜職・繁華街・深夜営業エリアの需要

昼食需要は、ビジネス街や住宅街で発生しやすい。

夜の夕食需要は、昼よりも比較的広いエリアで満遍なく発生しやすい。

深夜需要は地域差が大きい。夜職の人が多い繁華街エリアでは需要が出やすいが、住宅エリアでは深夜需要がほとんどない場合がある。

生活している人間の時間が違えば、食事の時間も変わる。その需要のズレを掴めれば、配達員は待機場所や移動方向を有利に選べる。

ピークタイムは回転率が重要。ロング案件は、たとえ単価が良くても次の短距離高回転案件を逃す可能性がある。

そのため、ピークタイムでは圧倒的に高単価でない限り、短距離で分単価に納得できる案件を優先しやすい。

一方、オフピークでは案件数が減るため、分単価がボーダー以上ならロング案件も受注候補になる。

### ロング案件の感覚

距離データがない現状では、所要時間やエリアの離れ方で代用する。

実務上の感覚:

- 僻地へ行く場合: 往復1時間以上
- 注文がありそうなエリアへ行く場合: 片道45分以上

同じロングでも、行き先が「鳴るエリア」か「戻りにくい僻地」かで意味が大きく変わる。

### good / normal / bad の評価基準

評価の大部分は分単価で決まる。

次点で以下が補正要素になる。

- 次につながる届け先
- 距離
- 店舗待機
- 戻りやすさ

したがって、評価分析では「good率」だけを見るより、分単価・行き先・待機時間との関係を見る方が価値がある。

### 稼げるエリアの基準

稼げるエリアは、基本的に受注地点を主軸に見る。

理由:

- 受注地点は、自分が実際に待機・稼働していたエリアである可能性が高い
- 配達完了地点は、自分の裁量というより運の要素が大きい

ただし、店舗の閉店や混雑状況を考える場合は、店舗地点も重要になる。

完了地点は「飛ばされ先」として補助的に見る。

### 店舗待機

理想は店舗到着後すぐ受け取り。

ただし、実害として分析する基準は以下が妥当。

- 5分以上: 注意
- 10分以上: 強い注意
- 15分以上: 避けたい / 調理待ちキャンセル級

平均待機時間だけでなく、5分超え率・10分超え率を見る方が実務に合う。

### 会社別分析

会社別分析は、単純に「どの会社が一番稼げるか」だけでは不十分。

見るべきこと:

- 時間帯ごとの平均単価
- 時間帯ごとの件数
- 時間帯ごとの時給
- 自分がどの会社を多く取っているか
- ピークタイムに特定の会社を見落としていないか
- 月ごと・季節ごとの変化

例:

- 夜ピークにRocketばかり取っていて、Uber Eatsや出前館を見ていない
- 閑散期は会社ごとの強弱が変わる
- 繁忙期は単価より件数・回転率が重要になる

## 再設計の基本方針

分析画面は「データの種類別」ではなく「配達員の判断別」に整理する。

## 推奨する分析カテゴリ

### 1. 今日どう動くか

目的:

- 今日の稼働中に判断する
- 続けるか、休むか、移動するか、切り上げるかを決める

統合候補:

- デイリーレポート
- 着地予測
- 直近3日の効率
- 今日の時間帯別件数
- 今日の会社別売上
- 今の時間の過去実績

表示例:

- このペースなら本日 ¥18,400 予測
- 18-21時は過去平均より強い時間帯
- 直近3日は夜ピークのロングで効率低下
- 今は続行推奨 / 移動推奨 / 休憩候補

### 2. この案件を取るか

目的:

- 受注判断に使う
- 条件ごとの合格ラインを育てる

統合候補:

- 効率化ポイント
- 会社別分析
- 平均単価
- 注文タイプ分析
- 天候 / 気象分析
- 低報酬トラップ
- ダブル・トリプル分析

重要な考え方:

- ピークとオフピークで受注基準を分ける
- ロング案件は時間帯によって評価を変える
- 会社を避けるのではなく、条件ごとの最低分単価を出す

表示例:

- 夜ピークのロングは ¥90/分以上なら合格
- オフピークのロングは ¥65/分以上なら許容
- この店舗は10分超え率が高いため、ピーク中は避ける
- ダブル案件はこの時間帯では ¥80/分以上だけ取る

### 3. どこで待つか / どこを避けるか

目的:

- 待機場所と避ける場所を決める
- 地図で感覚的に理解する

統合候補:

- 注文ヒートマップ
- 高単価配達マップ
- 店舗待機マップ
- エリア別分析

地図レイヤー案:

- 受注地点: 鳴る場所
- 高単価地点: 高単価が出た場所
- 店舗待機: 待たされる店
- 低効率地点: 分単価が低くなりやすい場所
- 完了地点: 飛ばされ先

基本は受注地点を主軸にする。

店舗地点は「待機リスク」「閉店・混雑リスク」を見るために使う。

完了地点は「飛ばされた結果」として参考扱いにする。

### 4. いつ働くか

目的:

- 稼働予定を決める
- 休む時間・働く時間を決める

統合候補:

- 時間帯分析
- 曜日分析
- 曜日 x 時間 時給表
- 推移（週次/月次）

表示例:

- 金曜18-21時は安定して強い
- 月曜14-17時は件数が少ない
- 深夜1-4時は地域によってデータ不足
- 今月は先月より夜ピークの単価が低下

曜日 x 時間表は価値が高いが、画面単体ではなく「いつ働くか」の中心機能として扱うとよい。

### 5. 月いくら増やせるか

目的:

- 改善効果を金額で見せる
- プレミアム価値にもつなげる

統合候補:

- 効率化ポイント
- 売上集計
- 目標逆算
- 月間改善見込み

表示例:

- 店舗待機リスクを避けると月 +¥6,800 見込み
- 夜ピークの低分単価ロングを減らすと月 +¥12,400 見込み
- 週1回、強い時間帯に2時間追加すると月 +¥18,000 見込み

## 新設する価値が高い分析

### ピーク別 受注基準分析

ピーク / オフピークで受注基準を分ける分析。

見る指標:

- 分単価
- 所要時間
- ロング判定
- 会社
- 注文タイプ
- 店舗待機
- 行き先エリア

目的:

- ピーク中に取ってよいロングの基準を出す
- オフピークで拾ってよい案件の基準を出す

### ロング機会損失分析

高単価に見えるロング案件でも、ピーク中に取ることで回転率が落ちていないかを見る。

見る指標:

- ピーク中のロング案件の分単価
- ロング後の待機時間
- ロング後に鳴るエリアへ戻れたか
- 同時間帯の短距離案件の分単価

目的:

- 「高単価だったが実は損だった」案件を見つける

### 店舗待機リスク分析

店舗ごとに待機リスクを可視化する。

見る指標:

- 平均待機時間
- 最大待機時間
- 5分超え率
- 10分超え率
- 15分超え率
- 調理待ちキャンセル数

目的:

- ピーク中に避ける店を把握する
- オフピークでも受けてよい店か判断する

### 会社の見落とし分析

特定の会社に偏りすぎていないかを見る。

見る指標:

- 時間帯 x 会社の件数
- 時間帯 x 会社の単価
- 時間帯 x 会社の時給
- 月別・季節別の変化

注意:

アプリは「見たが取らなかった案件」を記録していないため、本当の見落としは断定できない。

ただし、自分が実際に取っている会社の偏りは分析できる。

表示例:

- 夜ピークはRocket Now中心。Uber Eats / 出前館の比較データが少ない
- この時間帯は会社別判断に必要なデータが不足

### 基点・待機エリア分析

稼働エリアごとに、よく鳴る待機場所を基点として扱う。

考え方:

- 基点は固定ではない
- 都市やエリアによって複数存在する
- 全国のユーザー向けには、エリア名ベースの方が理解しやすい
- 需要はエリア属性によって時間帯が変わる
- ビジネス街、住宅街、繁華街では、強い時間帯が違う

将来的には「よく待機しているエリア」を自動検出し、そこを基点候補にしてもよい。

さらに発展させるなら、受注エリアを以下のような性質で推定できると分析価値が高い。

- 昼に強いエリア
- 夜に強いエリア
- 深夜にも需要があるエリア
- ピーク外でも一定需要があるエリア
- 戻りにくいエリア
- 待機には向かないエリア

## 既存分析の統合候補

### 時間帯分析 + 曜日分析 + 曜日 x 時間 時給表

統合先:

- いつ働くか分析

理由:

別画面で見るより、「稼働予定を決める」目的にまとめた方が行動に落とし込みやすい。

### 会社別分析 + 平均単価 + 効率化ポイント

統合先:

- この案件を取るか分析

理由:

会社別の単純比較より、条件付きの受注基準にした方が現場で使える。

### 天候分析 + 気象コンディション分析

統合先:

- この案件を取るか分析
- いつ働くか分析

理由:

天候そのものより、雨・風・気温によって受注基準や稼働判断がどう変わるかが重要。

### 注文ヒートマップ + 店舗待機マップ + 高単価配達マップ + エリア別分析

統合先:

- どこで待つか / どこを避けるか分析

理由:

地図を分けすぎると、ユーザーが頭の中で重ね合わせる必要がある。

レイヤー切り替えの方が直感的。

## UX方針

### 分析名は目的ベースにする

データ名ではなく、ユーザーの行動に近い名前にする。

例:

- 今日どう動く
- 案件判断
- 待機場所
- 稼働時間
- 月の改善

### 最初に結論を出す

グラフより先に、短い結論を表示する。

例:

- 今週は夜ピークが強い
- この店舗はピーク中に避けたい
- ロングはオフピークなら許容
- 金曜夜は優先稼働

### 数字は行動に変換する

悪い例:

- 平均時給 ¥1,420
- 平均単価 ¥680

良い例:

- 夜ピークは ¥80/分未満のロングを避ける
- この店は10分超え率が高いのでピーク中は避ける
- 火曜昼は弱いので短時間稼働にする

### 初心者にもプロにも使える階層にする

初心者:

- 今日のおすすめ
- 避けた方がいい条件
- 稼ぎやすい時間

プロ:

- 条件別分単価
- ピーク別ロング基準
- 会社別・月別比較
- 店舗待機リスク
- エリア別の細かい傾向

## 外部統計で確認したいこと

ピークタイムの初期値は、ユーザー実感として以下を採用する。

- 昼ピーク: 11:00-13:00
- 夜ピーク: 18:00-21:00
- 深夜ピーク: 01:00-04:00

ただし、地域差があるため、外部統計で補助的に確認する。

調べる候補:

- 生活時間統計における昼食・夕食時間
- 広島市中心部の時間帯別人流
- 深夜営業飲食店の分布
- Uber Eats等で深夜注文可能な店舗の分布
- 地域別に深夜ピークが成立するか

外部統計はアプリにそのまま入れるのではなく、初期設定や仮説の材料として使う。

最終判断は、ユーザー自身の実データと現場感覚で行う。

## 今後確認したい論点

### ピークタイムの地域別扱い

深夜ピークを全国共通にするのは危険。

候補:

- 深夜ピークを設定でON/OFF
- 実データが一定件数以上ある地域だけ自動表示
- 初期値はOFF、実績があれば表示
- 住宅街では深夜ピークを表示しない
- 繁華街・夜職エリアでは深夜ピークを候補にする
- 昼ピークはビジネス街・住宅街の両方で見る
- 夜ピークは広めのエリアで見る

ピーク判定は、全国一律の時間帯だけでなく、エリアごとの受注実績で補正するのが望ましい。

### ロング判定の実装方法

現状は距離データがないため、以下で代用する。

- 所要時間
- 受注地点と完了地点の直線距離
- 完了エリアが鳴るエリアかどうか
- 僻地エリアかどうか

将来的にはルート距離や移動時間推定を入れられると精度が上がる。

### 基点エリアの扱い

基点は固定ではなく、稼働エリアごとに異なる。

候補:

- よく受注しているエリアを自動で基点候補にする
- ユーザーが「よく待つ場所」を登録できるようにする
- ただし入力項目が増えるとUXが悪化するため、自動検出優先

### ミッション・クエストの扱い

ミッションやクエスト中は受注基準が変わる。

ただし、手入力項目を増やすとUXが悪化する。

現時点では、ミッションを個別入力させるより、以下のような扱いが現実的。

- 日次インセンティブは既存データとして扱う
- ミッション中の判断基準は明示的には入力させない
- 分析上は「日次インセンティブが大きい日」として後から補正的に見る

## 優先順位

### 優先度A

- 既存分析の目的別再編
- ピーク / オフピーク別の受注基準
- 店舗待機リスクの強化
- 地図分析のレイヤー統合

### 優先度B

- ロング機会損失分析
- 会社の見落とし分析
- 月別・季節別の会社比較
- 基点エリアの自動検出

### 優先度C

- 外部統計を使ったピーク初期値の検証
- 深夜ピークの地域別ON/OFF
- ルート距離・移動時間推定

## 初回実装メモ

2026-05-30 時点の初回実装では、まず既存分析を壊さずに「使って判断する」方向へ寄せた。

実装したこと:

- 分析メニューの順番を、今日どう動く → 案件判断 → 受注エリア → 店舗待機リスク → 稼働時間 → 会社別 → 売上集計、の順へ変更
- 「効率化ポイント」を「案件判断」に改名
- 案件判断画面に、時間帯別の受注ボーダーを追加
- ピーク定義を、昼ピーク 11-13時、夜ピーク 18-21時、深夜ピーク 1-4時、その他をオフピークとして扱うように変更
- 45分以上をロング候補、60分以上を戻りコストが重い案件として暫定表示
- 店舗待機リスクとして、5分以上率・10分以上率・待機中央値を表示
- 時間帯ごとの会社比率と会社別分単価を表示し、会社選びの偏りを見えるようにした
- 無料版は直近3日を使った今日の判断、PRO版は全期間を使った精密な自分専用ルールとして区切る
- 無料画面内に、全期間データで何が増えるかを示すPRO誘導カードを追加
- 通常起動は無料版に戻し、開発環境のみ `?premium=1` で有料版を検証できるようにした
- 開発環境のみ `?fixture=empty|edge|diverse|large` で空データ・欠損データ・多様データ・大量データを読み込める検証フックを追加

まだ実装していないこと:

- 地図系分析のレイヤー統合
- ロング後に鳴るエリアへ戻れたかの判定
- 基点エリアの自動検出
- 深夜ピークの地域別ON/OFF
- 外部統計によるピーク初期値の検証

## まとめ

このアプリの分析は、今後「分析項目を増やす」よりも、「配達員の判断に近づける」方向に進めるべき。

中心になる価値は、以下の3つ。

1. 自分専用の受注基準を育てる
2. 待つ場所・避ける店を感覚的に理解する
3. 稼働時間を売上が上がる場所へ寄せる

そのためには、既存分析を目的別に統合し、ピークタイム・ロング案件・店舗待機・受注エリア・会社選びの癖を、現場で使える形に変換していく。
